# References

## 目录结构

```text
05_References/
├── 01_Inbox/       # 尚未判断或消化的剪藏
└── 02_Library/     # 已消化且原文本身值得长期保留的精选文章
```

不按 AI、教育、产品等主题建立目录。跨主题内容使用 `tags`、知识卡片和项目链接组织。

## 工作流程

```text
剪藏到 Inbox
  ↓
快速判断
  ├─ 无价值 → 删除
  ├─ 暂时不处理 → 留在 Inbox
  ├─ 服务明确项目 → 链接或移入项目
  └─ 值得消化 → 阅读、质疑并形成判断
                   ↓
               产生至少一种结果
                   ├─ 知识卡片
                   ├─ 项目决定
                   ├─ 具体行动
                   └─ 明确无需提炼
                   ↓
               原文值得长期保留？
                   ├─ 是 → Library
                   └─ 否 → 删除
```

## Library 收录标准

原文本身至少符合一项，才进入 `02_Library/`：

- 论证完整，摘要无法替代
- 将来写作、课程或研究时可能引用
- 包含重要案例、数据、图表或一手证据
- 作者的表达本身值得反复学习
- 来源可能失效，需要保存本地副本
- 与长期关注方向高度相关

“读过”不是收录标准。普通文章提炼出有价值的内容后即可删除。

## 消化结果

消化不等于必须创建卡片。一篇剪藏只要产生以下任意结果，就可以结束处理：

- 创建一张稳定的知识卡片
- 改变某个项目判断
- 产生一个具体行动
- 留下可复用的证据或引用
- 经过判断，确认没有长期价值

可以使用：

- `reading-coach` — 理解、质疑、内化并行动
- `card-creator` — 将稳定洞察创建为知识卡片
- `critical-check` — 检查文章中的主张、证据与边界

## 文件与属性

建议文件名直接使用稳定、可识别的文章标题。日期、来源和主题放在 frontmatter：

```yaml
---
title: "文章标题"
author: "作者"
source_url: "https://example.com/article"
published_at:
clipped_at: "YYYY-MM-DD"
tags:
  - reference
---
```

文件所在目录已经表达处理状态，不再额外维护 `status` 属性。

## 版权与公开仓库

- 私人 Vault 可以按个人使用需求保存本地剪藏。
- 公开 Git 仓库不要提交未经许可的完整版权文章。
- 公开内容优先保留来源链接、少量必要引用和自己的理解。
- 如需在私人 Vault 保留全文，可将个人 Library 排除在公开同步之外。

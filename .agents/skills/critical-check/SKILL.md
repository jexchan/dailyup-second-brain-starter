# Critical Check: 观点质疑与证据校验

对指定内容进行“建设性怀疑”：拆出主张，检查证据，寻找反例，暴露隐含假设，并给出下一轮迭代方向。

这个技能的目标不是为了显得尖锐，也不是把内容打倒，而是帮助用户把一个观点理解得更深、更稳、更可迭代。

---

## 适用范围

使用本技能处理当前项目中的内容，包括：

- 用户直接贴出的段落、观点、文章草稿、课程设计、产品想法
- `04_Knowledge/00_Cards/` 中的指定卡片
- `03_Projects/` 中的项目判断、计划、目标、下一步
- `05_References/` 中的剪藏文章、摘要、证据材料
- `02_Daily/` 中的临时想法、阶段性判断

默认只读，不修改 vault。只有用户明确说“记录 / 更新 / 写回 / 保存到第二大脑”时，才按 AGENTS.md 的写回规则落盘。

---

## 核心姿态

### 1. 先理解，再质疑

先用自己的话复述内容的核心意思，确认它到底在主张什么。不要一上来挑刺。很多“漏洞”其实来自误读。

### 2. 区分主张类型

不同主张需要不同校验方式：

- **事实主张**：是否有来源？是否过时？是否需要外部检索？
- **因果主张**：有没有把相关性当因果？是否存在其他解释？
- **价值判断**：依赖哪些价值排序？是否有适用边界？
- **方法论主张**：在哪些场景有效？在哪些场景会失效？
- **产品/课程判断**：是否有用户、场景、约束和验证方式？
- **个人经验总结**：是否被单一经历过度外推？

### 3. 建设性怀疑

质疑要服务于迭代，而不是单纯否定。每个重要质疑后面最好接一个可行动问题：

- 需要补什么证据？
- 需要限定什么边界？
- 需要找什么反例？
- 需要改写哪一句？
- 下一步可以如何验证？

### 4. 标注确信度

把判断分成三类：

- **比较可靠**：已有材料或逻辑支撑较强
- **需要澄清**：表达可能成立，但边界、定义或对象不清
- **风险较高**：证据不足、推理跳跃、过度泛化或可能误导

## 工作流程

### Step 1: 读取项目规则

先读取 `AGENTS.md`，理解当前 vault 的目录结构、默认写作语言和写回规则。

如果任务涉及用户长期方向、受众或品牌表达，再按需读取：

- `01_Context/About_Me.md`
- `01_Context/Mission_and_Vision.md`
- `01_Context/Audience_Profiles.md`
- `01_Context/Brand_Voice.md`
- `01_Context/Current_Priorities.md`

如果任务涉及写作质量，再读取：

- `00_System/Writing_Rules.md`

### Step 2: 定位待校验内容

根据用户输入判断内容来源：

- 如果用户直接贴了内容：直接分析这段内容
- 如果用户给了文件路径：读取对应文件
- 如果用户给了 wikilink 或卡片名：优先在 `04_Knowledge/00_Cards/` 中搜索
- 如果用户只给主题：先用 `rg` 搜索相关文件，再说明本次采用的材料范围

搜索时优先使用：

```bash
rg -n "关键词|同义词|英文名" 04_Knowledge 03_Projects 05_References 02_Daily 01_Context
```

不要把一次搜索说成“全库没有”。如果没找到，应说“当前检索未发现”。

### Step 3: 提取主张地图

把内容拆成：

- **中心主张**：这段内容最想证明什么？
- **关键子主张**：支撑中心主张的判断有哪些？
- **证据/例子**：作者用了哪些事实、案例、经验、引用？
- **隐含假设**：哪些前提没有说出来，但论证依赖它们？
- **目标对象**：它到底适用于谁、什么场景、什么阶段？

如果内容很长，优先挑出最影响结论的 3-7 个主张。

### Step 4: 逐项质疑与校验

从以下角度检查：

#### 证据强度

- 证据是亲身经验、案例、专家观点、研究、数据，还是直觉判断？
- 证据是否足以支撑结论的强度？
- 证据是否只支持“可能相关”，却被写成“必然因果”？
- 是否需要最新资料、一手来源或外部事实核查？

当用户要求事实核查、最新信息、外部证据，或主张明显依赖现实变化时，应联网检索，并在回答中标注来源。否则优先使用 vault 内材料。

#### 推理链条

- 是否存在跳步？
- 是否混淆了概念层级？
- 是否把个体经验外推到普遍规律？
- 是否把必要条件说成充分条件？
- 是否把工具效果归因给方法论本身？

#### 反例与边界

- 在什么情况下这个观点不成立？
- 哪类人、任务、阶段、组织、学习场景不适用？
- 有没有 vault 中已有材料能构成反例或补充？
- 这个观点如果被极端化，会导致什么误解？

#### 替代解释

- 同一个现象是否可能由其他原因解释？
- 是否有更简单的解释？
- 是否存在被忽略的变量、激励、环境或选择偏差？

#### 表达清晰度

- 核心概念是否定义清楚？
- 论点和例子是否混在一起？
- 语气是否过满、过绝对？
- 哪些句子应该改成更可检验、更有边界的表述？

### Step 5: 生成迭代建议

输出不止指出问题，还要给出下一轮改进方向：

- 可以补充的证据
- 可以加入的反例
- 可以收窄的边界
- 可以重写的关键句
- 可以进一步追问的问题
- 可以沉淀为新卡片的主题

如果用户要进一步写成文章、课程、产品或卡片，可建议组合使用：

- `card-creator`：把关键反思沉淀为卡片
- `knowledge-gap`：继续诊断还缺什么材料

## 输出原则

- 使用中文，除非用户明确要求英文
- 先复述再质疑，避免误伤原意
- 明确区分“内容中已有证据”“基于材料的推断”“你的建议”
- 不要假装已经事实核查；没有查证就说“尚未查证”
- 不要为了平衡而制造牵强反对意见
- 不要只说“需要更多证据”，要说明需要哪类证据
- 对有价值但证据不足的观点，用“保留但降级表达”的方式处理
- 语气清晰、直接、建设性，不嘲讽，不居高临下

## 写回规则

默认不写回。

如果用户明确要求保存：

- 对单张卡片的质疑与修订建议 → 可追加到原卡片的“反思 / 迭代 / 待验证”部分，或新建一张 `question` / `insight` 卡片
- 对文章草稿的质疑 → 写入同目录的评论稿或修订建议文件
- 对项目判断的质疑 → 写入对应项目的 `Project.md`
- 对长期方法论的修订 → 建议先输出修改方案，再征求确认后写入 `04_Knowledge/`

写回前先检查附近是否已有相近文件，避免重复主题。

## 与其他技能的边界

- 需要综合一个主题的分散材料：用 `synthesis`
- 需要诊断为了写作/课程/产品还缺什么材料：用 `knowledge-gap`
- 需要把内容沉淀成新卡片：用 `card-creator`
- 需要从 Daily 中提炼长期知识：用 `evergreen-refactor`
- 需要追踪一个观点如何演变：用 `trace`
- 需要连接两个主题：用 `connect`
- 需要对某个观点、证据、推理、边界进行质疑和校验：用本技能

## 触发示例

**用户**：质疑一下这张卡片：[[insight_学习不是输入而是重组]]  
**应做**：读取卡片，拆出主张和证据，指出隐含假设、反例、边界和可改写版本。

**用户**：这个观点站得住吗？“AI 会让学习变成一种即时反馈系统。”  
**应做**：区分事实主张和方法论主张，检查因果链、适用边界和需要补充的证据。

**用户**：帮我 red team 一下这篇文章草稿。  
**应做**：先复述文章中心论点，再按证据、推理、反例、表达四个层面提出建设性质疑。

**用户**：帮我校验这个课程设计里的假设。  
**应做**：识别学习者、目标、练习、评估方式中的假设，指出最需要验证的地方。
